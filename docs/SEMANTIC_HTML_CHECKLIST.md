# 시맨틱 HTML 개선 체크리스트

프로젝트 전반의 시맨틱 HTML 적용을 위해 정리한 개선 항목입니다.

---

## 현재 시맨틱 적용 상태 (요약)

### ✅ 잘 적용된 부분

| 위치 | 적용 내용 |
|------|-----------|
| **MobileWrapper** | `Column as="main"`으로 본문만 main에 포함, Header/Menu는 main 밖 |
| **Menu** | 내부에 `Column as="nav"`, 상위에 `role="menu"` + `aria-label="사용자 메뉴"` |
| **PageTitle** | 제목을 `<h2>`로 렌더링, 뒤로가기 버튼 `aria-label` |
| **BottomButton.Container** | `role="group"`으로 주요 액션 영역 표시 |
| **Error** | `Column as="section"` + `role="alert"`, 제목 `<h2>` |
| **CourseDetail** | `Column as="article"`, 강의 제목 `<h1>` |
| **CourseListToolbar** | 툴바 `role="toolbar"` + `aria-label`, 정렬 영역 `Row as="section"` + `role="group"` |
| **SelectableList** | Container `as="section"` + `role="list"` + `aria-label`, Item `role="listitem"` |
| **layout** | `<html lang="ko">` |
| **폼** | SignIn/SignUp/CourseCreate 모두 `<form>` 사용 |

### ⚠️ 개선 여지

| 위치 | 현재 | 제안 |
|------|------|------|
| **app/layout.tsx** | body 직속에 `role="application"` | 체크리스트 권장대로 **제거**하고 그냥 `div`로 두기 (보조 기기 동작이 더 예측 가능) |
| **Header** | `PaddingHorizontalOverrideContainer`(div)에 `role="banner"`만 사용 | 시맨틱 태그로 **`<header>`** 감싸기 (또는 Container에 `as="header"` 지원) |
| **Header** | 사이트명 "COURSE"가 `<h2>` | 페이지마다 한 개의 최상위 제목이 있으려면 **`<h1>`** 로 두는 선택 가능 (현재도 h2로 동작 가능) |

### 정리

- **랜드마크**: main, section, article, role="menu"/"toolbar"/"alert"/"group" 등이 적절히 쓰이고 있음.
- **제목 계층**: PageTitle h2, CourseDetail h1, Header는 h2(또는 h1로 통일 가능).
- **폼·버튼**: form, button, aria-label 사용 적절.
- **남은 선택 작업**: 루트의 `role="application"` 제거, Header를 `<header>` 요소로 감싸기, 사이트명 h1/h2 결정.

## 1. 루트 레이아웃 (`app/layout.tsx`)

| 현재                                             | 제안                                 | 이유                                           |
| ------------------------------------------------ | ------------------------------------ | ---------------------------------------------- |
| `<div className="flex flex-col...">` (body 직속) | 유지 또는 `<div role="application">` | 앱 전체 래퍼. 시맨틱 랜드마크는 하위에서 처리. |

**두 제안의 차이**

- **유지** (지금처럼 그냥 `<div>`): 별도 역할 없이 레이아웃용 래퍼. 보조 기기는 이 div를 “랜드마크”로 인식하지 않고 건너뛰고, **자식 안의 `<header>`, `<main>`, `<nav>` 등**으로 문서 구조를 파악함. 일반적인 웹앱에서는 이 방식이 안전하고, 랜드마크는 MobileWrapper 내부(Header, Main 등)에서만 쓰면 됨.
- **`role="application"`**: “이 영역 전체가 애플리케이션(위젯)이다”라고 선언하는 ARIA 역할. 보조 기기가 **문서 모드가 아니라 애플리케이션 모드**로 이 영역을 다룰 수 있어, 키보드/포커스 동작이나 읽기 방식이 달라질 수 있음. 전체 앱 껍데기에 붙이기보다는, 특정 위젯(예: 캘린더, 에디터) 단위로 쓰는 것이 보통 권장됨.

**정리**: body 직속 래퍼는 **그냥 `div` 유지**하는 쪽을 권장하고, 시맨틱/랜드마크는 `MobileWrapper` 내부에서만 구분하는 것이 좋음.

## 2. 헤더 (`shared/components/ui/Header.tsx`)

| 현재                                           | 제안           | 이유                                           |
| ---------------------------------------------- | -------------- | ---------------------------------------------- |
| `PaddingHorizontalOverrideContainer` → `<div>` | **`<header>`** | 페이지/사이트 상단 헤더이므로 `<header>` 사용. |

**수정 방향**:

- `Header.tsx`에서 최상위를 `<header>`로 감싸거나,
- `PaddingHorizontalOverrideContainer`에 `as` prop을 두어 `as="header"`로 렌더링하도록 확장.

## 3. 메인 콘텐츠 영역 (`shared/components/container/MobileWrapper.tsx`)

| 현재                       | 제안         | 이유                                                         |
| -------------------------- | ------------ | ------------------------------------------------------------ |
| `<Column>` (children 래퍼) | **`<main>`** | 각 페이지의 주요 콘텐츠가 들어가는 영역이므로 `<main>` 사용. |

**수정 방향**:

- children을 감싸는 `Column`을 `<main>`으로 바꾸거나,
- `Column`에 `component`/`as` prop을 두어 `as="main"` 지원.

**주의**: 한 문서에 `<main>`은 하나만 두는 것이 좋음. 레이아웃이 이미 단일 메인 구조이므로 여기 한 번만 사용하면 됨.

## 4. 메뉴 (`shared/components/ui/Menu.tsx`)

| 현재                                                 | 제안                             | 이유                                                                                                 |
| ---------------------------------------------------- | -------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `PaddingHorizontalOverrideContainer` + `role="menu"` | **`<nav>`** + `role="menu"` 유지 | 로그아웃·테마 등 사용자 액션 메뉴. ARIA `role="menu"`는 유지하고, 영역 의미를 위해 `<nav>`로 감싸기. |

**수정 방향**:

- 최상위를 `<nav aria-label="사용자 메뉴">`로 감싸고, 기존 `role="menu"`는 내부 유지.

| --------------------------------------- | ----------------------------------------------------------------------- | -------------------------------------- |
| `Column` + `<h2>` + `<div>` (버튼 래퍼) | **`<section role="alert">`** 또는 **`<section aria-live="assertive">`** | 오류 메시지 영역을 알림 영역으로 명시. |

## 13. Flex 컴포넌트 (Row, Column)

| 현재         | 제안                     | 이유                                                                                                        |
| ------------ | ------------------------ | ----------------------------------------------------------------------------------------------------------- |
| `<div>` 기반 | **선택적으로 `as` prop** | 레이아웃 전용이므로 기본은 `div` 유지. 필요한 곳에서만 `as="main"`, `as="section"` 등으로 시맨틱 태그 지정. |

**수정 방향**:

- `Column`/`Row`에 `as?: keyof JSX.IntrinsicElements` 같은 prop을 두고, 지정된 경우 해당 태그로 렌더링.

## 적용 우선순위 제안

1. **높음**: `<main>` (MobileWrapper), `<header>` (Header), `<article>` (CourseDetail), PageTitle → 제목 태그(`h1`/`h2`).
2. **중간**: `<nav>` (Menu), `<section>` (강의 목록, 에러), Error `role="alert"`.
3. **낮음**: BottomButton을 `footer`/`role="group"`, PaddingHorizontalOverrideContainer `as` prop, Row/Column `as` prop.

## 참고: `role`이 하는 역할 (실제로 어떤 차이를 만드는지)

`role`은 **ARIA(Accessible Rich Internet Applications)** 속성으로, “이 요소가 **무슨 역할**인지”를 보조 기술(스크린 리더, 보이스 오버 등)에게 알려줍니다. 브라우저가 그냥 그린 화면에는 보이지 않고, **접근성 API**를 통해 전달됩니다.

### 1. 누가 사용하나

- **스크린 리더** (VoiceOver, NVDA, JAWS, TalkBack 등): 시각 장애인 사용자가 페이지를 “들으면서” 탐색할 때 사용.
- **보조 기술 전반**: 키보드만 쓰는 사용자, 포커스/랜드마크 이동 도구 등.

### 2. 실제로 어떤 상황에서 차이가 나나

| 상황               | role/시맨틱 없을 때                                                                                 | role/시맨틱 있을 때                                                                                                                                             |
| ------------------ | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **랜드마크 이동**  | “다음 블록”, “div” 같이만 읽힘. 헤더/메인/네비를 구분 못 함.                                        | “배너”, “메인 콘텐츠”, “네비게이션”처럼 **역할 이름**으로 이동 가능. “메인으로 건너뛰기” 같은 단축키로 바로 이동.                                               |
| **페이지 첫 방문** | 위에서부터 줄 단위로 전부 읽기 시작. 헤더·메뉴 다 읽고 나서야 본문 도달.                            | “배너”, “네비게이션”, “메인” 등으로 **구역 단위**로 이동해, 본문만 먼저 들어갈 수 있음.                                                                         |
| **커스텀 위젯**    | `<div onclick="...">` 는 그냥 “그룹” 또는 “블록”으로만 인식. 클릭 가능한지, 선택됐는지 알기 어려움. | `role="button"`, `role="tab"`, `role="menu"` 등으로 **동작 방식**이 전달됨. “버튼”, “탭”, “메뉴”로 읽히고, 키보드 동작도 기대에 맞게 동작하도록 연결할 수 있음. |
| **에러/알림**      | 에러 메시지가 화면 한쪽에 있어도, 포커스가 다른 곳에 있으면 읽히지 않을 수 있음.                    | `role="alert"` 이면 **포커스 위치와 상관없이** 곧바로 읽어 줌.                                                                                                  |

### 3. 시맨틱 태그와의 관계

- **`<header>`, `<main>`, `<nav>`, `<button>`** 같은 **시맨틱 HTML**을 쓰면, 브라우저가 이미 그에 맞는 **암시적(implicit) role**을 붙여 줍니다.  
  예: `<header>` → `role="banner"`, `<main>` → `role="main"`, `<nav>` → `role="navigation"`, `<button>` → `role="button"`.
- 그래서 **시맨틱 태그를 쓰면 같은 역할의 `role`을 따로 적지 않아도** 보조 기기가 같은 정보를 얻습니다.
- **`role`을 직접 쓰는 경우**:
  - 디자인/구조상 `<div>`나 `<span>`을 써야 하는데, **의미상** 버튼/탭/메뉴/배너 등인 경우.
  - 예: `<div role="button" tabindex="0">` → “이 div는 버튼처럼 써라”라고 알려주는 것.

### 4. 정리

- **role** = “이 요소를 **무슨 것으로 취급할지**”를 보조 기술에게 알려주는 힌트.
- **실제 차이** = 스크린 리더 사용자가 **랜드마크로 빠르게 이동**하거나, **에러를 바로 듣거나**, **커스텀 UI를 버튼/메뉴처럼** 사용할 수 있게 됨.
- 가능하면 **시맨틱 태그**(`<header>`, `<main>`, `<nav>`, `<button>` 등)를 쓰고, 어쩔 수 없이 div/span일 때만 해당 의미에 맞는 **role**을 붙이는 방식이 좋습니다.

## 참고: 시맨틱 요소 요약

| 요소          | 용도                                     |
| ------------- | ---------------------------------------- |
| `<header>`    | 사이트/페이지 상단 (로고, 메뉴 버튼 등)  |
| `<nav>`       | 내비게이션/메뉴 영역                     |
| `<main>`      | 문서당 하나, 주요 콘텐츠                 |
| `<article>`   | 독립적인 콘텐츠 (강의 상세, 카드 등)     |
| `<section>`   | 주제별 구역 (목록, 필터, 에러 메시지 등) |
| `<footer>`    | 페이지/문서 하단 (주 액션 바 포함 가능)  |
| `<h1>`~`<h6>` | 제목 계층 (한 페이지에 h1 하나 권장)     |

이 체크리스트를 기준으로 단계적으로 적용하면 시맨틱 HTML을 준수한 프로젝트로 정리할 수 있습니다.
